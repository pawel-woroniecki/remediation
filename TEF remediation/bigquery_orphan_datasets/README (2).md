# Orphan BigQuery Datasets Report

Scans all datasets in a BigQuery project, reads `BQ_DATASET_NAMES` from every locally cloned repo in a subgroup, and flags datasets that exist in BigQuery but are not referenced by any repo's `.gitlab-ci.yml`.

Answers questions like:
- Which datasets have no matching GitLab CI reference?
- Which objects exist inside those unmatched datasets?
- Which repos declare `BQ_DATASET_NAMES` and which do not?

---

## Script

`unmatched_bq_datasets_report.py`

---

## Outputs

Each run writes results to three destinations:

### 1. Local CSV files (under `--output-dir`)

Timestamped folder, e.g. `outputs/ndl_core_unmatched_bq_datasets_20260514T120000Z/`:

| File | Contents |
|---|---|
| `report.md` | Dataset-level summary |
| `unmatched_datasets.csv` | One row per unmatched dataset |
| `unmatched_objects.csv` | Object-level detail for unmatched datasets |
| `dataset_errors.csv` | Datasets whose metadata could not be queried |
| `repo_dataset_references.csv` | Repo-to-dataset mapping from `.gitlab-ci.yml` |

### 2. GCS bucket

All CSV files are uploaded to `gs://<gcs-bucket>/<gcs-prefix>/<execution_id>/`.

### 3. BigQuery (`devops_reports` dataset)

| Table | What is written |
|---|---|
| `orphan_datasets` | One row per dataset evaluated |
| `orphan_dataset_objects` | One row per object inside an orphan dataset |
| `executions` | One row recording the run outcome |

## How To Run

### Inside Cloud Run (standard)

Triggered automatically by `entrypoint.sh` as `orphan_datasets` report type. All required arguments are passed via the Cloud Run Job definition.

### Locally

```bash
python unmatched_bq_datasets_report.py \
  --gcp-project tefde-gcp-fastoss-prod \
  --reporting-project tefde-gcp-fastoss-dev \
  --workspace-root /workspace/repos/fastossb \
  --subgroup ndl_core \
  --gcs-bucket tefde-gcp-fastoss-dev-gcs-devops-reports
```

---

## Parameters

| Argument | Required | Default | Description |
|---|---|---|---|
| `--gcp-project` | yes | — | BigQuery project to scan for datasets |
| `--reporting-project` | no | `--gcp-project` | GCP project for the `devops_reports` BQ dataset |
| `--workspace-root` | yes | — | Parent folder containing subgroup directories |
| `--subgroup` | no | `ndl_core` | Subgroup folder under `--workspace-root` |
| `--git-ref` | no | `production` | Git branch used when reading `.gitlab-ci.yml` |
| `--output-dir` | no | `./outputs` | Root folder for local output |
| `--gcs-bucket` | no | *(skip upload)* | GCS bucket for CSV upload |
| `--gcs-prefix` | no | `orphan-datasets` | GCS path prefix |
| `--dataset-include-regex` | no | — | Only include matching datasets |
| `--dataset-exclude-regex` | no | — | Exclude matching datasets |

### Environment variables

| Variable | Description |
|---|---|
| `EXECUTION_ID` | Auto-generated by `entrypoint.sh` if not set. Can be overridden. |
| `OUTPUT_DIR` | Fallback for `--output-dir` |

---

## Matching Logic

1. List all datasets in the target BigQuery project
2. Scan every local Git repo under the selected subgroup
3. Read `BQ_DATASET_NAMES` from each repo's `.gitlab-ci.yml` at the selected Git ref
4. Split the variable by comma into concrete dataset names
5. Mark a dataset as orphaned if no repo references it

---

## Notes

- Datasets whose names start with `_` are excluded by default (hidden/temporary datasets).
- Hidden/system datasets that reject `INFORMATION_SCHEMA` queries are still listed in `dataset_errors.csv`.
- `EXECUTION_ID` is required. It is auto-generated by `entrypoint.sh` when running inside Cloud Run.
