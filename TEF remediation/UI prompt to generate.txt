## ✅ Prompt for Vibecoding AI

> **Context**
>
> I have an existing repository with **Python scripts** that generate DevOps and governance reports.
> These scripts:
>
> *   Are CLI‑based and already work correctly
> *   Require input parameters (CLI args + environment variables)
> *   Produce outputs as CSV files written to GCS and results written to BigQuery
> *   Must NOT be rewritten or have business logic changed
>
> The reports include:
>
> 1.  Commit‑based branch drift
> 2.  File‑diff‑based branch drift
> 3.  Environment vs code drift (Git vs BigQuery / GCS)
> 4.  Orphan BigQuery datasets (even if source code is missing)

***

> **Goal**
>
> Design and implement a **simple web UI** that allows users to:
>
> *   Select which report to run
> *   Provide the required input parameters
> *   Trigger execution of the existing scripts
> *   View execution status and links to outputs
>
> The UI should be a **thin orchestration layer**, not a reporting engine.

***

> **Architecture Constraints (Must Follow)**
>
> ✅ Scripts run inside a **Docker container** (Python 3.11 slim)
> ✅ One container image supports **all scripts**
> ✅ Scripts are executed as **Google Cloud Run Jobs** (batch, not long-running)
> ✅ Parameters are passed as **CLI arguments and environment variables**
> ✅ The GitLab PAT secret is stored in **GCP Secret Manager** and fetched at runtime by the container
> ✅ Execution is **stateless**
> ✅ Outputs are written to **GCS** (CSV files) and **BigQuery** (structured results)
> ✅ `EXECUTION_ID` is **auto-generated** by the container entrypoint (`uuidgen`) — the UI does not need to generate it
>
> 🚫 Do NOT embed business logic in the UI
> 🚫 Do NOT allow arbitrary shell execution
> 🚫 Do NOT hardcode secrets
> 🚫 Do NOT modify existing scripts

***

> **Execution Model**
>
> *   A wrapper entrypoint inside the container routes by report type:
>
>         run-report <report_type> [script arguments]
>
> *   Each report type maps to exactly one existing Python script.
> *   The UI triggers executions by invoking Cloud Run Jobs with arguments.
> *   The two GCP projects in use are:
>     *   `tefde-gcp-fastoss-dev-gke` — Cloud Run Jobs, GCS, Secret Manager
>     *   `tefde-gcp-fastoss-dev` — BigQuery dataset (`devops_reports`)
>
> Example Cloud Run Job invocation:
>
>     gcloud run jobs execute devops-reports-commit-drift \
>       --project=tefde-gcp-fastoss-dev-gke \
>       --region=europe-west3 \
>       --args="commit_drift,--gcp-project,tefde-gcp-fastoss-dev-gke,--reporting-project,tefde-gcp-fastoss-dev,--gcs-bucket,tefde-gcp-fastoss-dev-gcs-devops-reports"

***

> **UI Requirements**
>
> The UI should:
>
> *   Show a **list of available reports**
> *   Dynamically render a **parameter form** based on the selected report
> *   Validate required parameters before execution
> *   Trigger the corresponding Cloud Run Job
> *   Show:
>     *   Execution ID (read from BigQuery `executions` table or Cloud Run Job execution status)
>     *   Status (running / success / failed)
>     *   Link to GCS output folder
>
> The UI does NOT need advanced styling; clarity and correctness matter more.

***

> **Report Metadata Model (Important)**
>
> Reports should be defined declaratively, for example:
>
> ```yaml

>   label: "Commit-based branch drift"

>       type: string
>       required: true
>       default: "tefde-gcp-fastoss-dev-gcs-devops-reports"

>       type: string
>       required: false
>       default: "<today>"
>

>   label: "File-content branch drift"
>   cloud_run_job: "devops-reports-file-drift"

>       required: true

>       type: string
>       required: true

>       type: string

> ```
>
> The UI should use this metadata to generate forms automatically.

***

> **Technology Preferences**
>
> *   Backend: lightweight API (Python or Node.js)
> *   UI: simple web UI (React, Vue, or plain HTML is fine)
> *   Execution: Google Cloud Run Jobs (already provisioned via Terraform)
> *   Authentication: platform IAM or single sign‑on
>
> Use sensible defaults and explain assumptions.

***

> **Deliverables Expected**
>
> 1.  High‑level architecture diagram (textual description is fine)
> 2.  Backend API design (endpoints + payloads)
> 3.  UI behaviour and flow
> 4.  How parameters are passed to the Cloud Run Job
> 5.  How executions are tracked (BigQuery `executions` table + Cloud Run Job status)
> 6.  How GCS output links are surfaced to the user
>
> Focus on **clean separation of concerns, simplicity, and safety**.
